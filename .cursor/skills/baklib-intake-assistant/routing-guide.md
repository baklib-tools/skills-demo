# 数据放哪里：知识库、资源库与站点

供录入助手做**路由建议**与向用户说明理由。最终以用户确认与 MCP 实际能力为准。

**检索与用语一致**：建议按 [local-mirror.md](local-mirror.md) 将内容同步到 **`baklib_mirror_root` 下的 `知识库/`、`资源库/`、`站点/`**（Markdown），同步状态与 **DAM 标签/合集** 写入 **SQLite**；录入前优先查本地 `.md` 与库内标签记录，减少重复说法。

## 一、一句话对照


| 载体           | 最适合放什么                                                                  | 典型不适合                                  |
| ------------ | ----------------------------------------------------------------------- | -------------------------------------- |
| **知识库（KB）**  | 需要**阅读、检索、版本协作**的叙述性内容：说明文档、SOP、会议纪要文字稿、FAQ 正文、接口/配置说明、长文 Markdown/HTML | 原始大体积二进制「只存文件」为主（应放 DAM，KB 里**链接/嵌入**） |
| **资源库（DAM）** | 需**原样保管与复用**的文件：图片、PDF、音视频、安装包、设计稿、模板附件                                 | 以「段落说明」为主、且很少单独下载的纯文本（更适合 KB 文章正文）     |
| **站点（Site）** | **对外发布**的独立页面：帮助中心页、活动/落地页、需站点导航与模板的访客向内容                               | 纯内部沉淀、无对外发布需求（多放 KB）；仅归档文件（多放 DAM）     |


## 二、何时优先用站点（Site）

在以下情况**优先考虑站点页面**，而不是仅写 KB 文章：

- 需要在**官网/帮助站点导航**中出现固定入口、独立 URL 的页面  
- 内容强依赖**站点模板**（如固定版头版尾、组件变量 `template_variables`）  
- 与**营销活动、对外品牌**强绑定，读者是**外部访客**而非仅内部同事

若用户目标是「先记下来、团队内查」，默认倾向 **KB**；若明确「要上线到某某站点」，用 **Site**，并用 MCP 列出 `site_id`、父页面后再创建。

## 三、知识库：选哪个库、哪个「位置」

### 3.1 选知识库（`space_id`）

- 用 MCP **`kb_list_knowledge_bases`** 列出可选空间，**禁止编造**空间名称或 ID。  
- 按**业务域**选库（如产品 A / 部门 B / 客户 C），与用户约定一致即可写入记忆文件 `defaults.kb_space_id`。

### 3.2 树内位置（`parent_id` 与排序）

- Baklib 知识库文章支持**父子层级**：`parent_id` 指向**父文章**，常用于模拟「文件夹 / 栏目」。  
- **优先**在 **`{baklib_mirror_root}/知识库/`**（见 [local-mirror.md](local-mirror.md)）中浏览目录与 frontmatter，再用 **`kb_list_articles` / `kb_get_article`** 确认父级 **ID** 仍与线上一致。  
- **`position`**（若使用）用于同级排序；具体取值规则以接口与后台展示为准，不确定时先创建再按需 `kb_update_article` 调整。

### 3.3 建议决策

- **新主题、无合适父级**：与用户确认是「新建父级（一篇作为分类根）」还是「挂在现有某篇下」。  
- **多库都可放**：列出候选 `space_id` + 库名，说明各自适用场景，**请用户选一个**后再 `kb_create_article`。

## 四、资源库：放什么、如何补充「标签 / 合集」信息

### 4.1 适合进 DAM 的类型

- 图片 / 截图 / 图标  
- PDF、Office 文档、压缩包  
- 音视频  
- 其它需**下载原文件**或**在多处引用同一文件**的附件

上传用 **`dam_upload_entity`**（`file_path`、`type`、`name`）；必要时再用 **`dam_update_entity`** 更新展示名与说明。

### 4.2 标签与合集（SQLite + 目录，并与 MCP 能力对齐）

团队按 [local-mirror.md](local-mirror.md) 约定时：**合集**对应 **`资源库/<合集目录名>/`**；**标签及资源—标签关系**记在 **`sync_index_path` 的 SQLite** 中，并与线上定期对齐。录入助手查标签/合集：**以 SQLite 与现有文件夹名为准**（可选 YAML 导出仅作只读快照）。

当前公开的 Baklib MCP 工具描述中，**`dam_upload_entity` / `dam_update_entity` 仅体现**：本地上传、资源类型，以及更新 **`name`、`description`**。**未在工具参数中列出**「资源标签、合集 ID」等字段。

录入助手应这样处理用户关于「打标签、进合集」的诉求：

1. **用语先于操作**：从 SQLite（或导出的 `dam_taxonomy_path`）取**已有标签与合集名**，**逐字复用**；用户口述不一致时列出最接近项请其确认。  
2. **可自动做的**：上传后调用 **`dam_update_entity`**，在 **`description`** 中写入与库内一致的标签/合集说明，便于后台人工或后续同步对齐。  
3. **需人工或更强 API 的**：在 Baklib 控制台补全官方标签/合集（若 MCP 未覆盖）；未来 MCP 若增加参数，以**当时工具 schema** 为准，**勿臆造**。  
4. **向用户确认**：若要求全自动写标签/合集，先核对 MCP/API 是否支持；不支持则不要假装已写入。

### 4.3 「哪个合集」怎么定

- **有本地镜像时**：优先看 **`资源库/`** 下已有**合集文件夹名**与 SQLite 中的合集记录，推荐归入已有合集。  
- **MCP 未暴露合集参数时**：给出与 SQLite 一致的命名建议，并写入 `description`；由用户或同步程序在后台/下次同步对齐。  
- 团队可在记忆文件 `notes` 中约定「DAM 合集命名规范」，与 SQLite 中 `dam_collections`（或等价表）保持一致。

## 五、站点标签（Site tags）与 DAM 的区别

- MCP 提供 **`site_list_tags` / `site_create_tag`** 等，面向**站点**维度标签。  
- **不要默认**认为站点标签等于 DAM 资源标签；是否互通以 Baklib 产品与后台为准。  
- 页面与标签如何关联（若有）：以**当时 MCP 工具参数**与 Baklib 文档为准；当前 `site_update_page` 的公开 schema 未列出标签字段时，勿臆造参数，可提示用户在**站点后台**完成关联。

## 六、常见组合流程

1. **长文 + 多附件**：附件 → `dam_upload_entity`；正文 → `kb_create_article`，正文中引用 DAM 返回的标识（以 MCP 返回字段为准）。
2. **对外帮助页 + 下载包**：包进 DAM；站点页 `site_create_page` 中通过模板变量或正文链到资源。
3. **仅内部备忘**：通常 **KB 单篇**即可，无需站点发布。

## 七、歧义时的输出格式（给助手用）

向用户展示时建议包含：

- **推荐载体**（KB / DAM / Site）与**一条理由**  
- **具体落点**：`space_id` 或 `site_id`、拟使用的 **`parent_id`（若有）**、DAM 的 **`name` / `description` 要点**  
- 若涉及标签/合集但 MCP 无法写入：**单独一行**说明「后台需补一步」

用户确认后再调用写入类 MCP 工具。
